# Banco de Dados

## Aula 02 - 

## Projeto de banco de dados 

Para criar um banco de dados é necessário que seja elaborado um projeto. Como todo projeto ele é composto por etapas. 

A primeira etapa é onde acontece o levantamento de todas as funcionalidades e requisitos relacionados aos dados. Os projetistas de banco entrevistam os usuários e interessados no banco para compreender as suas necessidades. Ao final dessa etapa é feita a análise e um documento é gerado com o resutado de tudo que foi coletado.

Em seguida, os projetistas elaboram um modelo dos dados a partir do documento gerado na etapa anterior. Esse modelo é conhecido como modelo conceitual. O modelo conceitual descreve de forma concisa todas as necessidades do banco de dados sem entrar nos detalhes da implementação. Ele tem o objetivo de confirmar que todos os requisitos de dados foram atendidos. Ele é elaborado em uma notação de alto nível para facilitar a comunicação entre os projetistas do banco e as pessoas que tem interesse na sua implementação. Essa etapa é chamada de projeto conceitual. Com o modelo conceitual pronto, já é possível implementar o banco de dados utilizando um SGBD. Essa implementação é feita através da trasnformação do modelo conceitual em um modelo de implementação específico do SGBD. Essa etapa é chamada de projeto lógico. A última etapa do projeto é a elaboração de um projeto físico onde são especificadas as estruturas de armazenamento interno e a organização física dos arquivos do banco de dados. Nessa aula iremos detalhar a etapa de criação do modelo conceitual. Utilizaremos para isso um modelo de Entidade-Relacionamento (ER). Esse modelo descreve os dados como entidades, relacionamentos e atributos.

A entidade é o principal objeto do modelo ER. Ele representa algo físico como pessoas, carros, casas etc., mas também pode representar algo abstrato como empresas ou cursos de uma escola. As entidades definem os atributos que são necessários para que um elemento seja considerado daquela entidade. Por exemplo um entidade aluno pode ter os atributos nome, sobrenome, matrícula, data de nascimento etc.. Para que o elemento seja considerado um aluno ele deve possuir esses atributos. A entidade também é chamada de Tipo de Entidade. Já o conjunto desses elementos que possuem os mesmos atributos da entidade é chamado de conjunto de entidades. Cada elemento desse conjunto terá um valor para os atributos dessa entidade. No exemplo da entidade aluno, podemos ter um elemento do conjunto com os valores Fábio, Neves, 990203 e 20/08/1978 para os atributos nome, sobrenome, matrícula e data de nascimento da entidade aluno.

Um atributo pode ser simples ou atômico quando ele possui um valor que não pode ser dividido em outros atributos. Já atributos compostos podem ser divididos em mais atributos. Um exemplo seria o caso da entidade aluno possuir um atributo endereço. Esse atributo por sua vez poderá ser dividido em outros atributos como nome da rua, cep, número, bairro etc.. Um atributo pode ter um valor único ou ser multivalorado. Até agora vimos apenas valores únicos para os atributos, mas uma entidade poderia ter um atributo onde poderiam existir mais de um valor. Um exemplo seria uma entidade Despesa que pode ter um atributo categoria da despesa. Nesse caso, poderíamos ter uma despesa específica que possui duas categorias e outra despesa com apenas uma categoria. Atributos podem também ser calculados a partir de um outro atributo. É o caso da entidade aluno possuir um atributo idade que é calculado a partir da data de nascimento. Nesse caso, o atributo idade é chamado de atributo derivado. Os atributos que não são derivados são chamados de atributos armazenados. Atributos compostos e multivalorados podem ser combinados e formarem novos atributos. Nesse caso temos o que chamamos de atributos complexos. Quando um elemento de uma entidade possui um atributo que não tem um valor definido ou não conhecido, atribuimos um valor especial chamado de nulo. O valor nulo representa a ausência do valor daquele atributo.

As entidades possuem uma restrição de unicidade. Isso quer dizer que cada elemento do conjunto de entidades deve ser único e possuir uma identidade. Essa identidade é definida pelos valores de determinados atributos que combinados devem ser únicos para cada elemento do conjunto. Chamamos de chave o conjunto mínimo de atributos que definem essa restrição de unicidade. 

No modelo ER um elemento de uma entidade pode necessitar fazer uma referência a um outro elemento de uma outra entidade. Por exemplo um elemento da entidade Funcionário pode ter um atributo departamento que indica qual o departamento que o funcionário trabalha. No nosso modelo iremos dar um destaque especial a esse atributo e vamos chamá-lo de relacionamento e vamos representá-lo de uma forma destacada. Assim como acontece na entidade o relacionamento também é chamado de tipo de relacionamento. O tipo relacionamento possui um conjunto de relacionamentos que representam o conjunto de associações entre os elementos das entidades que participam desse relacionamento. 

Vamos estudar agora algumas definições dos relacionamentos. O grau do relacionamento diz respeito ao número de entidades que participam de um relacionamento. Um auto relacionamento, por exemplo, é de grau 1 ou unário porque só um entidade participa do relacionamento. O relacionamento entre duas entidades é de grau 2 ou binário e assim por diante. Já a razão da cardinalidade entre uma relação binária mostra a quantidade de elementos que podem existir em cada lado da relação. Por exemplo a entidade Departamento pode ter uma relação com a entidade Funcionário (relacionamento binário) e uma razão de cardinalidade de 1:N. Isso que dizer que um elemento da entidade Departamento pode se relacionar com até N elementos da tabela funcionário, mas o elemento da entidade funcionário só tem relação com um elemento da entidade departamento. Da mesma forma podemos ter a razão de cardinalidade 1:1 e a razão de cardinalidade N:M. Por último temos a definição de restrição de existência. Essa restrição pode ser total ou parcial. Na existência total ela obriga o elemento de uma entidade a ter um relacionamento com uma outra entidade (a sua entidade proprietária). Um exemplo dessa retrição seria o caso dos elementos da entidade funcionário existirem apenas se possuirem um relacionamento com a entidade departamento indicando o local de trabalho. Também podemos dizer que todos os elementos que pertencem a entidade funcionários possuem um relacionamento com a entidade proprietária departamento. Já na existência parcial não existe essa obrigatoriedade.

Um exemplo especial da restrição de existência total é quando a entidade possui uma chave formada pela combinação da chave da entidade proprietária e algum atributo próprio. Nesse caso a entidade é chamada de entidade fraca pois seus atributos não são capazes de formar uma chave sem a combinação com a chave da entidade proprietária. Note que a entidade fraca sempre possui um relacionamento com uma restrição de exsitência total, porém o inverso não é verdade. Podemos ter uma entidade com uma chave própria que pode ter uma restrição de existência total com outra entidade.

Ainda sobre os relacionamentos podemos ter relacionamentos que possuem atributos. Por exemplo no caso do relacionamento de um funcionario trabalhar em um departamento podemos ter um atributo chamado início do trabalho que possui o valor da data em que o funcionário começou a trabalhar em determinado departamento.
